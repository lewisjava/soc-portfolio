# SIEM Fundamentals

A Security Information and Event Management (SIEM) system does what the name implies — it's an event and security log management system, core to SOC analyst work.

## Logs, logs, and more logs

Even in a relatively small network, a massive volume of logs gets generated.

![Log volume feeding into a SIEM](/static/images/notes/detection-monitoring/log-volume-siem.png)

**Host-centric log sources** — events occurring on a host (Linux, Windows, servers, etc.):

- A user accessing a file
- A user attempting to authenticate
- Process execution activity
- A process adding/editing/deleting a registry key or value
- PowerShell execution

**Network-centric log sources** — from hosts communicating with each other or the internet, generated by devices like firewalls, IDS/IPS, and routers:

- An SSH connection
- A file accessed via FTP
- Web traffic
- A user accessing company resources via VPN
- Network file sharing activity

## Features of a SIEM

![Core SIEM features](/static/images/notes/detection-monitoring/siem-features.png)

- **Centralised log collection** — collects logs from all sources into one place, solving the problem of scouring individual devices for logs.
- **Normalisation** — formats logs, which usually differ device to device, and breaks them down for ease of understanding (parsing).
- **Correlation** — finds relationships between logs from different sources. For example: a user logs in via VPN from a never-before-seen IP, accesses documents on a shared drive, executes a PowerShell script, and the system makes an outbound network connection. Individually, each activity looks fine — but correlated together, they could point to data exfiltration resulting from compromised VPN credentials.
- **Real-time alerting** — SIEM detects malicious activity based on its rules. Many come built-in by default, but analysts can also create new detection rules.
- **Dashboards and reporting** — where data is presented for analysis: alert highlights, system notifications, health alerts, failed login attempts, events ingested count, rules triggered, top domains visited.

## Log sources

- **Windows** — logs everything in Event Viewer.
- **Linux** — stores logs in directories:
  - `/var/log/httpd` — HTTP request/response and error logs
  - `/var/log/cron` — cron job events
  - `/var/log/auth.log` and `/var/log/secure` — authentication-related logs
  - `/var/log/kern` — kernel-related events
- **Web server** — Apache logs commonly write to `/var/log/apache` or `/var/log/httpd`.

## Log ingestion methods

- **Agent / Forwarder** — a lightweight tool (Splunk calls this a forwarder) installed on the endpoint, configured to capture and send important logs to the SIEM server.
- **Syslog** — a widely used protocol to collect data from various systems and send real-time data to a centralised destination.
- **Manual Upload** — some SIEMs (Splunk, ELK, etc.) allow offline data ingestion for quick analysis.
- **Port-Forwarding** — the SIEM listens on a certain port, and endpoints forward data to it directly.

## Alerting process and how rules get built

Detection rules are logical expressions set to trigger under specific conditions, for example:

- Five failed login attempts in 10 seconds → `Multiple Failed Login Attempts`
- Login successful after multiple failed attempts → `Successful Login After Multiple Login Attempts`
- A user plugs in a USB device (if restricted by policy)
- Outbound traffic exceeds 25MB → potential data exfiltration attempt

**Example — building a rule:**

Adversaries often clear event logs during the post-exploitation phase to remove their tracks. Windows Event ID 104 fires every time a user clears event logs. Rule: *if log source is WinEventLog and Event ID is 104 → trigger "Event Log Cleared" alert.*

**Another example:**

Adversaries commonly run `whoami` after privilege escalation. Useful fields for this rule: log source (WinEventLog), Event ID (4688 — process execution), and `NewProcessName`.

Rule: *if log source is WinEventLog AND EventCode is 4688 AND NewProcessName contains "whoami" → trigger "WHOAMI Command Execution Detected" alert.*

## Alert investigation outcomes

- Alert is a **false positive** — may require tuning the rule to avoid similar false positives recurring.
- Alert is a **true positive** — perform further investigation.
- Contact the asset owner to inquire about the activity.
- Suspicious activity confirmed — isolate the infected host.
- Block the suspicious IP.
